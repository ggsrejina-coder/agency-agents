# План реалізації агентної роботи (UA)

Цей документ описує практичний запуск **перших агентів** із цього репозиторію для реальних задач (MVP, фіча, редизайн, реліз).

## 1) Що є в репозиторії (коротко)

Репозиторій організований у доменні директорії, де кожен файл — окремий спеціаліст:

- `engineering/` — реалізація (frontend/backend/mobile/devops)
- `design/` — UX/UI, бренд, візуал
- `product/`, `project-management/` — пріоритезація, планування, трекінг
- `testing/` — валідація, докази, перевірки якості
- `marketing/`, `support/`, `specialized/`, `spatial-computing/` — доменні/нішеві агенти

## 2) Рекомендований стартовий стек агентів (перші 5)

Для більшості продуктів достатньо почати з такого складу:

1. **project-manager-senior** — декомпозиція задач, scope, пріоритети.
2. **agents-orchestrator** — координує пайплайн і handoff між агентами.
3. **Frontend Developer** або **Backend Architect** — основна реалізація.
4. **EvidenceQA** — перевірка із доказами (скріни/артефакти).
5. **testing-reality-checker** — фінальний quality gate перед релізом.

> Якщо у вас мобільний продукт — замініть **Frontend Developer** на **Mobile App Builder**.

## 3) Базова операційна модель (MVP-ready)

### Фаза A — Discovery
- PM агент збирає вимоги, фіксує "must-have"/"nice-to-have".
- Результат: backlog + порядок реалізації.

### Фаза B — Delivery Loop
- Orchestrator бере 1 задачу із backlog.
- Dev агент реалізує **тільки** цю задачу.
- **EvidenceQA** валідовує і дає PASS/FAIL.
- Якщо FAIL — повернення в розробку з конкретним фідбеком.

### Фаза C — Release Gate
- Reality Checker робить інтеграційну перевірку.
- Результат: "READY" або "NEEDS WORK" із чіткими блокерами.

## 4) Налаштування перших агентів (операційні інструкції)

Нижче — стандартні інструкції, які варто вставляти в старт кожного сеансу.

### 4.1 Інструкція для PM
```text
Ти project-manager-senior.
Побудуй task list із пріоритезацією (P0/P1/P2), залежностями та критеріями приймання.
Не додавай фічі поза вимогами.
Виводь задачі у форматі checkbox + Definition of Done.
```

### 4.2 Інструкція для Orchestrator
```text
Ти agents-orchestrator.
Працюй по одній задачі за раз: Dev -> QA Evidence -> рішення.
Максимум 3 retry на задачу.
Не переходь до наступної задачі без PASS поточної.
Веди короткий статус: поточна задача, спроба, ризики, ETA.
```

### 4.3 Інструкція для Dev-агента
```text
Реалізуй тільки поточну задачу із task list.
Після змін: короткий changelog, список файлів, ризики.
Не чіпай scope сусідніх задач.
```

### 4.4 Інструкція для QA Evidence
```text
Перевір тільки поточну задачу та її acceptance criteria.
Формат відповіді: PASS/FAIL, знайдені дефекти, докази, кроки відтворення.
За відсутності доказів став FAIL.
```

### 4.5 Інструкція для Final QA
```text
Виконай інтеграційну перевірку всіх P0/P1 задач.
Фінальний висновок: READY або NEEDS WORK.
Для NEEDS WORK обов'язково дай blocker list.
```

## 5) Мінімальні артефакти, які треба вести в проєкті

- `project-tasks/<project>-tasklist.md`
- `project-docs/<project>-decision-log.md`
- `testing/<project>-qa-evidence.md`
- `testing/<project>-release-check.md`

## 6) KPI для оцінки ефективності агентів

- `% задач, що пройшли QA з 1-ї спроби`
- `Середня кількість retry на задачу`
- `Lead time задачі (від start до PASS)`
- `Кількість blocker-інцидентів на реліз`

## 7) Коли масштабувати команду агентів

Додавайте нових спеціалістів, якщо:

- backlog > 20 задач або > 2 паралельні потоки;
- є суттєвий UX/бренд контур (додаємо `design-*` агентів);
- реліз включає growth/ASO/SMM складову (додаємо `marketing-*`);
- потрібні BI/звітність/операції (додаємо `support-*` і `specialized/*`).
