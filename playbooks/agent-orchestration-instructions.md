# Інструкції з оркестрації агентів

Нижче — практичні правила, які можна використати як базову операційну політику для multi-agent роботи в цьому репозиторії.

## Операційні принципи

1. **Один власник етапу**: на кожному етапі тільки один агент відповідає за фінальний результат.
2. **Явний контракт handoff**: кожен агент передає далі `Контекст → Що зроблено → Докази → Ризики → Наступний крок`.
3. **Ніякого "тихого" scope creep**: якщо вимоги не покриті — створюється окрема задача.
4. **Quality gate обов'язковий**: без PASS від QA етап не вважається завершеним.
5. **Рішення на основі evidence**: фінальний вердикт має бути підкріплений артефактами.

## Рекомендована послідовність агентів

```text
Project Manager Senior
  -> UX Architect
    -> Senior Developer
      -> API Tester + Evidence Collector
        -> Reality Checker
```

## Шаблон handoff-повідомлення

```markdown
## Handoff
- Етап: <назва>
- Вхідні дані: <що отримано>
- Що виконано: <список>
- Артефакти: <посилання/файли>
- Ризики: <список>
- Блокери: <так/ні + деталі>
- Рекомендований наступний агент: <назва>
- Definition of Done виконано: <так/ні>
```

## Шаблон оцінки якості (для QA і Reality Checker)

```markdown
## Validation Report
- Scope: <що перевіряли>
- Критерії: <перелік acceptance criteria>
- Результати по критеріях:
  - [PASS|FAIL] <критерій 1> — <доказ>
  - [PASS|FAIL] <критерій 2> — <доказ>
- Регресії: <перелік або "немає">
- Ризик-рейтинг: <Low|Medium|High>
- Фінальний вердикт: <READY|NEEDS WORK>
- Обов'язкові правки перед релізом: <список>
```

## Політика ескалації

- Якщо один і той самий критерій провалено **2+ рази**, задача автоматично ескалується назад до PM для перегляду scope/criteria.
- Якщо QA не може зібрати докази (невідтворювано), статус задачі — `BLOCKED`, реліз блокується.
- Якщо є конфлікт між архітектурою та реалізацією, пріоритет має архітектурний контракт до узгодження змін.

## Мінімальний Definition of Done для першої ітерації

Задача вважається завершеною лише якщо:
- опис виконання зрозумілий non-technical стейкхолдеру;
- є технічний артефакт (код/конфіг/док);
- є QA evidence по acceptance criteria;
- Reality Checker підтвердив `READY` або дав вичерпний список `NEEDS WORK`.
